<objective>
**Phase Goal (from ROADMAP.md, verbatim):** When a Hermes session — primary or subagent — completes a turn, a Hermes lifecycle hook deterministically writes a marker record at `~/.hermes/state/revenium/markers/<sid>.jsonl` with a meaningful `task_type` and `operation_type`, independent of whether the agent loaded the `revenium` skill or executed the FINAL ACTION self-classification code. Token attribution by activity in Revenium becomes observable without depending on agent compliance.

This is a single fat plan covering all 10 HOOK-* requirements. The components are tightly coupled — HOOK.yaml + handler.py + test fixtures + setup-local.sh + setup.md + test methods all ship together as one coherent migration. Splitting risks intermediate states where the hook is partially installed (gateway loads a manifest with no handler, or a handler with no install path). Same shape as Phase 3's single-fat-plan; same D-02-style commit-friendly task ordering.

The implementer ships in 15 atomic commits, each leaving the full test suite green. Three of the 18 locked decisions in CONTEXT.md were re-litigated by RESEARCH.md against the live Hermes source — the planner has folded those findings into the task actions below:

1. **C1/C2 wording fix**: `hermes hooks list` / `hermes hooks test` are for SHELL hooks declared in `~/.hermes/config.yaml::hooks`, NOT for the Python event-hook system targeted by Phase 6. Operator verification uses gateway startup logs + direct Python invocation. ROADMAP.md SC1/SC2 wording is non-binding (the underlying behavior is the same); CONTEXT.md D-02 still locks the file layout — the only change is the verification *mechanism*.

3. **D-03 resolved**: handler imports `agent.auxiliary_client.call_llm` (in-process sync) and wraps with `asyncio.to_thread()`. No direct httpx fallback in v1.

D-17 and D-18 are load-bearing backward-compat invariants: SKILL.md FINAL ACTION block is UNCHANGED; Phase 3 cron pipeline is UNCHANGED. This plan touches NEITHER file. Double-write is prevented by D-13 (tail-check at 30s wall-clock proximity) in the handler.

        """Walk state.db.sessions.parent_session_id chain. Returns input sid if it has
        no parent. Read-only URI mode prevents WAL lock contention with Hermes writer.
        Depth-capped to defeat pathological corrupted parent chains."""
        try:

    ```python
    def _recent_marker_pair_exists(sid: str, within_seconds: float = 30.0) -> bool:
        """D-13: return True if the marker file's tail carries a GUARDRAIL+CHAT pair
        whose most recent ts is within `within_seconds` of time.time(). Used to skip
        the hook write when the agent's SKILL.md FINAL ACTION snippet already wrote
        markers for this turn. Per Pitfall 6 option (a) — wall-clock proximity."""
        marker_path = MARKERS_DIR / f"{sid}.jsonl"
        if not marker_path.is_file():
            return False
        try:
            lines = marker_path.read_text(encoding="utf-8").splitlines()
        except OSError:
            return False
        # Walk backward, collect the most recent valid record's ts and operation_type.
        # We need to see BOTH a GUARDRAIL AND a CHAT within the window to consider it
        # a complete agent-written pair.
        now = time.time()
        seen_ops = set()
        for line in reversed(lines):
            try:
                rec = json.loads(line)
            except json.JSONDecodeError:
                continue
            ts = rec.get("ts")
            op = rec.get("operation_type")
            if not isinstance(ts, (int, float)) or not isinstance(op, str):
                continue
            if (now - ts) > within_seconds:
                break  # records are timestamp-ordered (append-only); no point continuing
            if op in ("GUARDRAIL", "CHAT"):
                seen_ops.add(op)
                if seen_ops >= {"GUARDRAIL", "CHAT"}:
                    return True
        return False
    ```
    NOTE: this assumes markers are appended in roughly timestamp order, which is the invariant Phase 2 + Phase 6 both maintain (each writer atomically appends two records with `time.time()` at write time). Out-of-order records would defeat the early-break optimization but not the correctness — the function would just keep scanning until the for-loop ends.
