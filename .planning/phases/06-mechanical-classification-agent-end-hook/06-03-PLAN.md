But a CLI substantive turn still produced no marker file. Root cause: `classifier._count_tools_in_current_turn(sid)` reads `~/.hermes/sessions/<sid>.jsonl` to count tool calls in the current turn. `hermes_cli/oneshot.py::_create_session_db_for_oneshot` writes session data only to `~/.hermes/state.db`, not to the per-session JSONL the gateway uses — so for CLI one-shot sessions the JSONL is absent, the count returns 0, and combined with the empty `response_preview` (the on_session_end payload doesn't include response text) BOTH heuristic-skip conditions match → silent no-op.

**Locked decision (D-20, in 06-CONTEXT.md gap-closure addendum #2):** Switch the tool-count source of truth from the per-session JSONL to `state.db.sessions.tool_call_count`. JSONL becomes a fallback. state.db is the universal source — `tool_call_count` is populated for every session source (CLI, gateway, interactive, ACP, cron) because it's a column on the `sessions` table written by every code path that runs `run_conversation`. The classifier already opens state.db via `sqlite3.connect(..., mode=ro)` for the parent-session walk in `_walk_to_root_session`; adding one parameterized SELECT against the same row is the minimum-blast-radius fix.
