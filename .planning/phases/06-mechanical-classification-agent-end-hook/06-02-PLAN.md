**Locked decision (D-19, in 06-CONTEXT.md gap-closure addendum):** Replace the gateway-hook integration with a `hermes_cli` plugin that registers `on_session_end` — emitted from `hermes-agent/run_agent.py:15164` for EVERY `run_conversation()` call (gateway-served + CLI + interactive + ACP + cron-spawned). Reference pattern: `hermes-agent/plugins/disk-cleanup/` ships `plugin.yaml` + `__init__.py` with `register(ctx)` calling `ctx.register_hook("on_session_end", handler)`.

**Why a plugin and not a hook:** Hermes has two distinct event registration mechanisms. Gateway hooks (YAML manifest under `~/.hermes/hooks/`) listen to gateway-bus events. Plugins (Python `register(ctx)` under `~/.hermes/plugins/`) listen to CLI/agent-core bus events. The event we want (`on_session_end`) lives on the other bus. We have to switch registration mechanisms.

**This plan is the SECOND plan in Phase 6 (06-02-PLAN.md) and runs in the gap-closure execution wave** (`--gaps-only` invocation). The first plan (06-01) shipped the gateway-hook tree; this plan supersedes it. The shared-module refactor lands BEFORE the gateway-hook deletion so every commit leaves the full test suite green.

**Adds HOOK-11** to REQUIREMENTS.md and ROADMAP.md: "Universal session coverage — the classifier fires on every Hermes session end regardless of source (gateway-served, CLI one-shot, interactive, ACP, cron), via the `hermes_cli` plugin system's `on_session_end` event."

**Out of scope (deliberately, per CONTEXT.md gap-closure addendum):**
- Keeping the gateway hook alongside the plugin (would double-fire and require dedup; single-fire is the cleaner invariant — delete the old hook outright).
- Editing upstream `hermes-agent/` source (we do not own that codebase).
- Cleaning up `~/.hermes/hooks/revenium-classifier/` on operator machines (stale-but-harmless under the new plugin; documented in setup.md).
