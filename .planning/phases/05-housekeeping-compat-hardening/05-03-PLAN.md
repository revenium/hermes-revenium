**Why now / why this shape:**
- **D-34 (WR-01):** The pipe-row protocol added by Phase 4 (11 fields including `m_agent` + `m_trace`) currently relies on a comment-only contract that future upstream writers will not include `|`, `\n`, or `\r` in marker `agent`/`trace_id` values. Today's classifier and SKILL.md FINAL ACTION snippet never populate these fields (D-23 explicitly defers writer-side changes), but Phase 6's plugin architecture could allow third-party writers (e.g., `revenium-skill-name = "my-skill|v2"` from a maliciously named skill). The 3-line sanitization at the Python emission boundary is defense in depth that costs nothing in the universal-empty case and prevents a subtle desync bug in pathological cases.
- **D-35 (WR-02):** The `local row` declaration at line 549 was a relic from a pre-11-pipe iteration of `hermes-report.sh::main` where rows were read into `row` first and then parsed. Phase 3 refactored to direct `while IFS='|' read -r ...` consumption; `row` was never re-removed. The variable is declared but never referenced. `set -u` does NOT bark on declared-but-unused locals, but the dead declaration is noise that misleads future readers. Single-line cleanup.
- **D-36 (WR-03):** Phase 4 added two test methods that construct `base_env` via `**os.environ` + a few explicit overrides. The overrides cover `HERMES_HOME` and `REVENIUM_STATE_DIR` but NOT `REVENIUM_MARKERS_DIR`, `REVENIUM_MARKERS_READY_DIR`, `REVENIUM_TAXONOMY_FILE` — three env vars that `common.sh` honors. A developer running tests with those vars exported in their shell could see the cron subprocess read/write against their REAL `~/.hermes/state/revenium` directory. WR-03 plugs the leak by adding the three overrides; the existing test pattern is preserved (additive, not refactored), so the diff stays minimal.
